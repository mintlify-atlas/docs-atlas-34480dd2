## Dev mode vs production mode

<CodeGroup>

```bash Dev mode
# Starts an in-memory, pre-unsealed, single-node server.
# All data is lost when the process stops.
vault server -dev

# Specify a fixed root token (convenient for local development)
vault server -dev -dev-root-token-id="root"

# Set a custom listen address
vault server -dev -dev-listen-address="0.0.0.0:8200"
```

```bash Production mode
# Requires at least one -config flag.
# The server starts sealed and must be initialized and unsealed.
vault server -config=/etc/vault/config.hcl

# Multiple configs are merged in the order they are provided
vault server -config=/etc/vault/config.hcl -config=/etc/vault/extra.hcl

# Point at a directory; all .hcl and .json files are loaded
vault server -config=/etc/vault/config.d/
```

</CodeGroup>

<Warning>
  Dev mode stores everything in memory and starts pre-unsealed with a known root token. Never run dev mode in production. As stated in the source (`command/server.go`): "do not run 'dev' mode in production".
</Warning>

## Starting a production server

<Steps>
  <Step title="Write a configuration file">
    See the [configuration guide](/setup/configuration) for a complete reference. At minimum, you need a storage stanza and a listener stanza:

    ```hcl
    # /etc/vault/config.hcl
    storage "raft" {
      path    = "/vault/data"
      node_id = "vault-1"
    }

    listener "tcp" {
      address       = "0.0.0.0:8200"
      tls_cert_file = "/etc/vault/tls/vault.crt"
      tls_key_file  = "/etc/vault/tls/vault.key"
    }

    api_addr     = "https://vault.example.com:8200"
    cluster_addr = "https://vault.example.com:8201"
    disable_mlock = true
    ```
  </Step>

  <Step title="Start the server">
    ```bash
    vault server -config=/etc/vault/config.hcl
    ```

    The server prints its configuration on startup:

    ```
    ==> Vault server configuration:

           Administrative Namespace:
                       Api Address: https://vault.example.com:8200
                               Cgo: disabled
                   Cluster Address: https://vault.example.com:8201
              Environment Variables: ...
                        Go Version: go1.22.0
                        Log Level: info
                            Mlock: supported: true, enabled: true
                    Recovery Mode: false
                          Storage: raft (HA available)
                          Version: Vault v1.22.0-beta1

    ==> Vault server started! Log data will stream in below:
    ```
  </Step>

  <Step title="Initialize Vault">
    Initialization is a one-time operation that prepares the storage backend and generates the root key. It only needs to be run on one node in an HA cluster.

    Set the environment variable to point at your server:

    ```bash
    export VAULT_ADDR='https://vault.example.com:8200'
    ```

    Run initialization with the default parameters (5 key shares, threshold of 3):

    ```bash
    vault operator init
    ```

    To customize the number of key shares and threshold:

    ```bash
    vault operator init \
      -key-shares=5 \
      -key-threshold=3
    ```

    The output contains the unseal keys and the initial root token:

    ```
    Unseal Key 1: WcMCKLHurCPkj3Uz8JuEWlDrMnlKVKuLME6E4GQDS5Rp
    Unseal Key 2: X4NRMSqjgYT/dRH5nkBnXneSWaFfzDG8SqG8n6v0hkFY
    Unseal Key 3: cJKlHOTT8INrZJGQ5FHoeTl6g+JNHQjQq3aHoQ9Fbe2P
    Unseal Key 4: G+U5MN+NJYQZMezOqxKv+1vbMHdLFlhTKT4fOIPBpbPy
    Unseal Key 5: A5Dj9YY2Cw+ICnJvZMeGRzUkWrYeKW5ZXDfFlXVr1GhS

    Initial Root Token: hvs.XXXXXXXXXXXXXXXXXXXXXXXX

    Vault initialized with 5 key shares and a key threshold of 3.
    Please securely distribute the key shares printed above.
    ```

    <Warning>
      Save the unseal keys and root token in secure, separate locations immediately. Vault does not store the root key. If you lose more key shares than `(total - threshold)`, the encrypted data is permanently inaccessible.
    </Warning>

    The defaults are defined in `command/operator_init.go`:
    - `defKeyShares = 5`
    - `defKeyThreshold = 3`
  </Step>

  <Step title="Unseal Vault">
    After initialization (and after every restart), Vault is sealed. Provide at least `key-threshold` unseal keys, one at a time:

    ```bash
    vault operator unseal
    # Prompt: Unseal Key (will be hidden): <paste key 1>

    vault operator unseal
    # Prompt: Unseal Key (will be hidden): <paste key 2>

    vault operator unseal
    # Prompt: Unseal Key (will be hidden): <paste key 3>
    ```

    Each call shows the current unseal progress. After the threshold is reached:

    ```
    Key             Value
    ---             -----
    Seal Type       shamir
    Initialized     true
    Sealed          false
    Total Shares    5
    Threshold       3
    Version         1.22.0-beta1
    Storage Type    raft
    Cluster Name    vault-cluster-...
    HA Enabled      true
    ```

    You can also pass the key as an argument, but this is not recommended because it will appear in shell history:

    ```bash
    vault operator unseal IXyR0OJnSFobekZMMCKCoVEpT7wI6l+USMzE3IcyDyo=
    ```

    To discard previously entered keys and start over:

    ```bash
    vault operator unseal -reset
    ```

    <Info>
      The `vault status` exit code reflects seal state: `0` = unsealed, `1` = error, `2` = sealed.
    </Info>
  </Step>

  <Step title="Log in with the root token">
    ```bash
    vault login
    # Prompt: Token (will be hidden): <paste root token>
    ```

    Or pass the token directly:

    ```bash
    vault login hvs.XXXXXXXXXXXXXXXXXXXXXXXX
    ```

    Or set it as an environment variable:

    ```bash
    export VAULT_TOKEN='hvs.XXXXXXXXXXXXXXXXXXXXXXXX'
    ```

    <Tip>
      The root token has unrestricted access. Generate more narrowly scoped tokens or use auth methods (AppRole, Kubernetes, AWS IAM, etc.) for applications and users. Revoke the root token once initial setup is complete.
    </Tip>
  </Step>
</Steps>

## Running as a systemd service

For production Linux deployments, run Vault as a systemd service so it restarts automatically on failure and integrates with the system journal.

```ini
# /etc/systemd/system/vault.service
[Unit]
Description=HashiCorp Vault - A tool for managing secrets
Documentation=https://developer.hashicorp.com/vault/docs
Requires=network-online.target
After=network-online.target
ConditionFileNotEmpty=/etc/vault/config.hcl
StartLimitIntervalSec=60
StartLimitBurst=3

[Service]
Type=notify
Environment=VAULT_ADDR=https://127.0.0.1:8200
Environment=VAULT_SKIP_VERIFY=false
User=vault
Group=vault
ProtectSystem=full
ProtectHome=read-only
PrivateTmp=yes
PrivateDevices=yes
CapabilityBoundingSet=CAP_SYSLOG CAP_IPC_LOCK
AmbientCapabilities=CAP_IPC_LOCK
NoNewPrivileges=yes
ExecStart=/usr/bin/vault server -config=/etc/vault/config.hcl
ExecReload=/bin/kill --signal HUP $MAINPID
KillMode=process
KillSignal=SIGINT
Restart=on-failure
RestartSec=5
TimeoutStopSec=30
StartLimitInterval=60
LimitNOFILE=65536
LimitMEMLOCK=infinity

[Install]
WantedBy=multi-user.target
```

Enable and start the service:

```bash
sudo systemctl daemon-reload
sudo systemctl enable vault
sudo systemctl start vault
sudo systemctl status vault
```

View logs:

```bash
journalctl -u vault -f
```

<Info>
  Vault notifies systemd when the server is ready using `sd_notify`. This requires the `Type=notify` in the unit file and `dumb-init` or a compatible init system.
</Info>

## Checking server status

```bash
vault status
```

This command (`command/status.go`) prints the current seal and HA status. It works even when the server is sealed.

| Exit code | Meaning |
|-----------|----------|
| `0` | Unsealed |
| `1` | Error |
| `2` | Sealed |
