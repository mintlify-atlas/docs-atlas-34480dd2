---
title: "Okta Auth Method"
description: "The Okta auth method authenticates users with Okta, with support for Okta MFA."
---

The Okta auth method allows users to authenticate using their Okta credentials. Vault verifies the credentials against the Okta API and assigns policies based on Okta group membership.

## Enabling Okta auth

```bash
vault auth enable okta
```

## Configuration

```bash
vault write auth/okta/config \
  org_name="dev-123456" \
  api_token="00AaBbCcDd..." \
  base_url="okta.com" \
  token_ttl="1h" \
  token_max_ttl="24h"
```

### Configuration parameters

<ParamField path="org_name" type="string" required>
  Name of the Okta organization, used to construct the Okta API URL. This is the subdomain of your Okta URL. For `https://dev-123456.okta.com`, the `org_name` is `dev-123456`.
</ParamField>

<ParamField path="api_token" type="string">
  Okta API key for accessing the Okta API. Used to verify group memberships. If not set, group policy mappings will not work.
</ParamField>

<ParamField path="base_url" type="string" default="okta.com">
  The base domain for the Okta API. Defaults to `okta.com`. Use `oktapreview.com` for preview environments.
</ParamField>

<ParamField path="bypass_okta_mfa" type="boolean" default="false">
  When true, requests by Okta for an MFA check will be bypassed. This also disables certain status checks on the account, such as whether the password is expired. Use with caution.
</ParamField>

<ParamField path="token_policies" type="string[]">
  List of policies applied to all tokens generated by this auth method, in addition to any group- or user-specific policies.
</ParamField>

<ParamField path="token_ttl" type="string">
  The incremental lifetime for generated tokens.
</ParamField>

<ParamField path="token_max_ttl" type="string">
  The maximum lifetime for generated tokens.
</ParamField>

<ParamField path="token_bound_cidrs" type="string[]">
  List of CIDR blocks restricting which IPs can authenticate and use the resulting token.
</ParamField>

<ParamField path="token_period" type="string">
  If set, tokens will be periodic with the given period.
</ParamField>

<ParamField path="token_type" type="string" default="default">
  The type of token to generate: `service`, `batch`, or `default`.
</ParamField>

<Warning>
  Enabling `bypass_okta_mfa` also prevents Vault from detecting expired passwords and other account states checked by Okta during MFA. Authentication will succeed even if the Okta account has issues.
</Warning>

## Mapping Okta groups to policies

Policies are assigned based on Okta group membership. The API token must be set for group lookups to work.

```bash
# Map an Okta group to a policy
vault write auth/okta/groups/engineering policies=engineering-policy

# Map multiple policies
vault write auth/okta/groups/devops policies=ops-policy,secrets-admin

# List configured group mappings
vault list auth/okta/groups

# Read a group mapping
vault read auth/okta/groups/engineering

# Delete a group mapping
vault delete auth/okta/groups/engineering
```

## Mapping users to policies

In addition to group policies, you can assign policies directly to specific Okta users:

```bash
vault write auth/okta/users/user@example.com \
  policies=admin-policy \
  groups=engineering

vault list auth/okta/users
```

## Logging in

```bash
vault login -method=okta username=user@example.com
```

You will be prompted for your password. If Okta MFA is configured, you will also be prompted to complete the MFA challenge.

Supply the password inline (not recommended for production scripts):

```bash
vault login -method=okta username=user@example.com password=secretpassword
```

## MFA support

Vault supports Okta's MFA mechanisms. When a user logs in and Okta requires MFA, Vault will prompt the user to complete the Okta MFA challenge.

Supported Okta MFA factors:

- **Okta Verify** (push notification or TOTP)
- **Google Authenticator** (TOTP)
- **SMS** authentication
- **Email** authentication
- **FIDO2/WebAuthn** (via Okta)

<Steps>
  <Step title="User submits credentials">
    The user calls `vault login -method=okta username=user@example.com`.
  </Step>
  <Step title="Vault verifies password with Okta">
    Vault submits the username and password to the Okta API.
  </Step>
  <Step title="Okta requests MFA">
    If MFA is required, Okta sends a push notification or returns a challenge.
  </Step>
  <Step title="User completes MFA">
    The user approves the push notification or enters the TOTP code.
  </Step>
  <Step title="Vault issues a token">
    After MFA is satisfied, Vault returns a token with the resolved policies.
  </Step>
</Steps>

## Example: Full setup

```bash
# 1. Enable
vault auth enable okta

# 2. Configure with org and API token
vault write auth/okta/config \
  org_name="mycompany" \
  api_token="00Axxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx" \
  token_ttl="8h" \
  token_max_ttl="24h"

# 3. Map Okta groups to policies
vault write auth/okta/groups/vault-admins policies=admin
vault write auth/okta/groups/developers policies=developer
vault write auth/okta/groups/operations policies=ops

# 4. Log in
vault login -method=okta username=alice@mycompany.com
```

## Read the current configuration

```bash
vault read auth/okta/config
```

<Note>
  The `api_token` is never returned when reading the configuration. You must re-set it if it needs to be changed.
</Note>
