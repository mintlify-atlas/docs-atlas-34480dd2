---
title: "vault server"
description: "Start a Vault server. Supports production configuration-file mode and in-memory development mode."
---

## Overview

The `vault server` command starts a Vault server that responds to API requests. By default, Vault starts in a sealed state and must be initialized and unsealed before it can serve requests.

```
Usage: vault server [options]
```

<Warning>
  Do not run `vault server -dev` in production. Dev mode stores data in memory, starts unsealed, and uses a fixed root token.
</Warning>

## Examples

<Tabs>
  <Tab title="Production mode">
    ```bash
    vault server -config=/etc/vault/config.hcl
    ```
  </Tab>
  <Tab title="Dev mode">
    ```bash
    vault server -dev -dev-root-token-id="root"
    ```
  </Tab>
  <Tab title="Dev mode with TLS">
    ```bash
    vault server -dev-tls -dev-tls-cert-dir=/tmp/vault-tls
    ```
  </Tab>
  <Tab title="Multiple config files">
    ```bash
    vault server -config=/etc/vault/config.hcl -config=/etc/vault/extra.hcl
    ```
  </Tab>
</Tabs>

## Command Options

<ParamField path="-config" type="string">
  Path to a configuration file or directory of configuration files. This flag can be specified multiple times to load multiple configurations. If the path is a directory, all files ending in `.hcl` or `.json` are loaded.
</ParamField>

<ParamField path="-exit-on-core-shutdown" type="bool" default="false">
  Exit the Vault server process if the Vault core shuts down.
</ParamField>

<ParamField path="-recovery" type="bool">
  Enable recovery mode. In this mode, Vault is used to perform recovery actions. Using a recovery token, the `sys/raw` API can be used to manipulate storage.
</ParamField>

<ParamField path="-experiment" type="string">
  Name of an experiment to enable. Experiments should **not** be used in production, and associated APIs may have backwards incompatible changes between releases. This flag can be specified multiple times. Can also be set via the `VAULT_EXPERIMENTS` environment variable as a comma-separated list.
</ParamField>

<ParamField path="-pprof-dump-dir" type="string">
  Directory where generated pprof profiles are written. If left unset, files are not generated.
</ParamField>

## Log Flags

<ParamField path="-log-level" type="string">
  Log verbosity level. Supported values: `trace`, `debug`, `info`, `warn`, `error`. Overrides the `VAULT_LOG_LEVEL` environment variable.
</ParamField>

<ParamField path="-log-format" type="string">
  Log output format. Supported values: `standard`, `json`. Overrides the `VAULT_LOG_FORMAT` environment variable.
</ParamField>

<ParamField path="-log-file" type="string">
  Path to the file Vault should use for logging.
</ParamField>

<ParamField path="-log-rotate-bytes" type="int">
  Number of bytes a log file should reach before being rotated.
</ParamField>

<ParamField path="-log-rotate-duration" type="duration">
  Duration after which a log file should be rotated.
</ParamField>

<ParamField path="-log-rotate-max-files" type="int">
  Maximum number of archived log files to keep.
</ParamField>

## Dev Options

<ParamField path="-dev" type="bool">
  Enable development mode. Vault runs in-memory and starts unsealed. **Do not use in production.**
</ParamField>

<ParamField path="-dev-tls" type="bool">
  Enable TLS development mode. Vault runs in-memory and starts unsealed, with a generated TLS CA, certificate, and key. **Do not use in production.**
</ParamField>

<ParamField path="-dev-tls-cert-dir" type="string">
  Directory where generated TLS files are created when `-dev-tls` is specified. If unset, files are written to a temporary directory.
</ParamField>

<ParamField path="-dev-tls-san" type="string">
  Additional Subject Alternative Name (DNS name or IP address) to include in the generated certificate when `-dev-tls` is specified. Can be specified multiple times. The certificate always includes `localhost`, `localhost4`, `localhost6`, and `127.0.0.1` by default.
</ParamField>

<ParamField path="-dev-root-token-id" type="string">
  Initial root token to use in dev mode. Corresponds to the `VAULT_DEV_ROOT_TOKEN_ID` environment variable. Only applies in dev mode.
</ParamField>

<ParamField path="-dev-listen-address" type="string" default="127.0.0.1:8200">
  Address to bind to in dev mode. Corresponds to the `VAULT_DEV_LISTEN_ADDRESS` environment variable.
</ParamField>

<ParamField path="-dev-no-store-token" type="bool" default="false">
  Do not persist the dev root token to the token helper (usually the local filesystem). The token will only be displayed in command output.
</ParamField>

## Production Configuration File

In production mode, `-config` is required. A minimal HCL configuration looks like:

```hcl
storage "raft" {
  path    = "/opt/vault/data"
  node_id = "node1"
}

listener "tcp" {
  address     = "0.0.0.0:8200"
  tls_cert_file = "/etc/vault/tls/vault.crt"
  tls_key_file  = "/etc/vault/tls/vault.key"
}

disable_mlock = true
api_addr      = "https://vault.example.com:8200"
cluster_addr  = "https://vault.example.com:8201"
ui            = true
```

<Note>
  After starting a new Vault server for the first time, you must run `vault operator init` to initialize the server before any requests can be served.
</Note>
