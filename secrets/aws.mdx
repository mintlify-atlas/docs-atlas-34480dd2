---
title: "AWS Secrets Engine"
description: "The AWS secrets engine generates dynamic IAM credentials and STS tokens for AWS services."
---

The AWS secrets engine generates AWS access credentials (IAM users or STS tokens) on demand. Generated credentials are automatically revoked when their lease expires, eliminating the need to manually manage long-lived AWS access keys.

## Enabling the engine

```bash
vault secrets enable aws
```

## Configuring AWS credentials

Vault needs AWS credentials to create and manage IAM users or assume roles.

```bash
vault write aws/config/root \
  access_key=AKIAIOSFODNN7EXAMPLE \
  secret_key=wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY \
  region=us-east-1
```

<ParamField path="access_key" type="string">
  AWS access key with permission to create IAM users, attach policies, and assume roles.
</ParamField>

<ParamField path="secret_key" type="string">
  AWS secret key corresponding to the access key.
</ParamField>

<ParamField path="region" type="string">
  AWS region for API calls.
</ParamField>

<ParamField path="iam_endpoint" type="string">
  Custom IAM endpoint URL. Useful for testing with localstack or GovCloud.
</ParamField>

<ParamField path="sts_endpoint" type="string">
  Custom STS endpoint URL.
</ParamField>

<ParamField path="username_template" type="string">
  Go template for generated IAM usernames. Defaults to a `vault-<display_name>-<policy_name>-<timestamp>-<random>` format.
</ParamField>

<Tip>
  Instead of configuring static root credentials, you can run Vault on an EC2 instance or ECS task with an IAM role. Vault will use the instance metadata service automatically.
</Tip>

## Rotating root credentials

Vault can rotate the root access key it was configured with:

```bash
vault write -f aws/config/rotate-root
```

After rotation, the previous access key is deleted from AWS. Vault stores the new credentials.

## Credential types

The AWS secrets engine supports four credential types, defined in `builtin/logical/aws/path_roles.go`:

| Credential Type | Description |
|---|---|
| `iam_user` | Creates a new IAM user with attached policies |
| `assumed_role` | Calls `sts:AssumeRole` and returns temporary credentials |
| `federation_token` | Calls `sts:GetFederationToken` and returns temporary credentials |
| `session_token` | Calls `sts:GetSessionToken` using the root credentials |

## Creating roles

### IAM user role

Vault creates a new IAM user and attaches the specified managed policies:

```bash
vault write aws/roles/s3-read \
  credential_type=iam_user \
  policy_arns=arn:aws:iam::aws:policy/AmazonS3ReadOnlyAccess
```

With an inline policy document:

```bash
vault write aws/roles/s3-write \
  credential_type=iam_user \
  policy_document=-<<EOF
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Action": ["s3:PutObject","s3:GetObject","s3:DeleteObject"],
    "Resource": "arn:aws:s3:::my-bucket/*"
  }]
}
EOF
```

### Assumed role

Vault calls `sts:AssumeRole` on a pre-existing IAM role:

```bash
vault write aws/roles/deploy \
  credential_type=assumed_role \
  role_arns=arn:aws:iam::123456789012:role/DeploymentRole \
  default_sts_ttl=1h \
  max_sts_ttl=4h
```

### Federation token

```bash
vault write aws/roles/read-only \
  credential_type=federation_token \
  policy_arns=arn:aws:iam::aws:policy/ReadOnlyAccess \
  default_sts_ttl=15m \
  max_sts_ttl=1h
```

### Role parameters

<ParamField path="credential_type" type="string" required>
  One of `iam_user`, `assumed_role`, `federation_token`, or `session_token`.
</ParamField>

<ParamField path="role_arns" type="string">
  Comma-separated ARNs of IAM roles to assume. Only valid when `credential_type=assumed_role`.
</ParamField>

<ParamField path="policy_arns" type="string">
  Comma-separated ARNs of managed IAM policies. For `iam_user`, these are attached to the user. For `assumed_role` and `federation_token`, they act as a permission filter.
</ParamField>

<ParamField path="policy_document" type="string">
  JSON inline IAM policy. For `iam_user`, attached to the user. For STS types, passed as the `Policy` parameter to filter permissions.
</ParamField>

<ParamField path="iam_groups" type="string">
  Comma-separated IAM group names. Generated IAM users are added to these groups.
</ParamField>

<ParamField path="iam_tags" type="object">
  Key-value pairs of IAM tags to set on generated IAM users.
</ParamField>

<ParamField path="session_tags" type="object">
  Key-value pairs of session tags for `assumed_role` credentials.
</ParamField>

<ParamField path="default_sts_ttl" type="string">
  Default TTL for STS credential types when no TTL is explicitly requested.
</ParamField>

<ParamField path="max_sts_ttl" type="string">
  Maximum allowed TTL for STS credential types.
</ParamField>

<ParamField path="user_path" type="string" default="/">
  IAM path for the generated user. Only valid for `iam_user` credential type.
</ParamField>

<ParamField path="permissions_boundary_arn" type="string">
  ARN of an IAM policy to attach as a permissions boundary to generated IAM users.
</ParamField>

<ParamField path="external_id" type="string">
  External ID to pass when assuming the role. Only valid for `assumed_role`.
</ParamField>

```bash
vault read aws/creds/my-role
```

```bash
  --header "X-Vault-Token: $VAULT_TOKEN" \

```

## Lease duration and revocation

For `iam_user` credentials, the default lease is controlled by the mount's `default_lease_ttl` and `max_lease_ttl`. When the lease expires, Vault automatically deletes the IAM user and its associated access keys.

For STS credential types, the TTL is set at role creation time via `default_sts_ttl` and `max_sts_ttl`. STS credentials are non-renewable and expire server-side when their TTL elapses.

```bash
# Renew a lease
vault lease renew aws/creds/s3-read/f3e92392-7d9c-09c8-c921-575d62fe80d9

# Revoke a lease immediately
vault lease revoke aws/creds/s3-read/f3e92392-7d9c-09c8-c921-575d62fe80d9
```

## Static roles

For long-lived IAM users that Vault manages (rotates credentials on a schedule), use static roles:

```bash
vault write aws/static-roles/my-static-role \
  username=my-existing-iam-user \
  rotation_period=1h
```

Read the current credentials:

```bash
vault read aws/static-creds/my-static-role
```

<Note>
  Static roles manage an existing IAM user's access key. Vault rotates the key on the configured schedule. Unlike dynamic roles, static roles do not create or delete IAM users.
</Note>
