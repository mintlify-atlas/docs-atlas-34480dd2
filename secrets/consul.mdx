---
title: "Consul Secrets Engine"
description: "The Consul secrets engine generates dynamic Consul ACL tokens with configurable policies and TTLs."
---

The Consul secrets engine generates dynamic Consul ACL tokens on demand. Each token is created with the configured policies and roles, and is automatically revoked when its lease expires.

## Enabling the engine

```bash
vault secrets enable consul
```

## Configuring the Consul connection

Vault needs a Consul management token (or a token with sufficient ACL permissions) to create and revoke tokens.

```bash
vault write consul/config/access \
  address="127.0.0.1:8500" \
  token="<consul-management-token>"
```

### Connection parameters

<ParamField path="address" type="string" required>
  Address of the Consul server, in `host:port` format.
</ParamField>

<ParamField path="scheme" type="string" default="http">
  URI scheme for the Consul address. Use `https` for TLS-enabled Consul clusters.
</ParamField>

<ParamField path="token" type="string">
  Consul ACL token. Must have permission to create and revoke tokens. If not provided, Vault attempts to bootstrap the ACL system.
</ParamField>

<ParamField path="ca_cert" type="string">
  PEM-encoded CA certificate for verifying the Consul server's TLS certificate.
</ParamField>

<ParamField path="client_cert" type="string">
  PEM-encoded client certificate for mutual TLS with Consul.
</ParamField>

<ParamField path="client_key" type="string">
  PEM-encoded client key for mutual TLS with Consul.
</ParamField>

With TLS:

```bash
vault write consul/config/access \
  address="consul.example.com:8501" \
  scheme=https \
  token="<management-token>" \
  ca_cert=@/etc/consul/ca.pem
```

## Creating roles

Roles define the policies and metadata attached to tokens generated by Vault.

### Role with Consul policies

```bash
vault write consul/roles/my-role \
  consul_policies="read-only,monitoring" \
  ttl=1h \
  max_ttl=24h
```

### Role with Consul roles

```bash
vault write consul/roles/service-role \
  consul_roles="service-discovery" \
  ttl=30m
```

### Role with service identities

```bash
vault write consul/roles/web-role \
  service_identities="web:dc1" \
  ttl=1h
```

### Role with node identities

```bash
vault write consul/roles/node-role \
  node_identities="server-1:dc1" \
  ttl=24h
```

### Role parameters

<ParamField path="consul_policies" type="string[]">
  Comma-separated list of Consul ACL policy names to attach to the generated token. Required for Consul 1.4+ when `token_type=client`.
</ParamField>

<ParamField path="consul_roles" type="string[]">
  Comma-separated list of Consul ACL role names to attach to the generated token. Available in Consul 1.5+.
</ParamField>

<ParamField path="service_identities" type="string[]">
  List of Consul service identity strings in the format `<service_name>:<datacenter>`. Available in Consul 1.5+.
</ParamField>

<ParamField path="node_identities" type="string[]">
  List of Consul node identity strings in the format `<node_name>:<datacenter>`. Available in Consul 1.8.1+.
</ParamField>

<ParamField path="ttl" type="string">
  Default TTL for generated tokens.
</ParamField>

<ParamField path="max_ttl" type="string">
  Maximum TTL for generated tokens.
</ParamField>

<ParamField path="local" type="boolean" default="false">
  If set, the generated token is local to the current datacenter and is not replicated globally. Available in Consul 1.4+.
</ParamField>

<ParamField path="consul_namespace" type="string" default="default">
  Consul namespace for the generated token. Available in Consul 1.7+.
</ParamField>

<ParamField path="partition" type="string" default="default">
  Consul admin partition for the generated token. Available in Consul 1.11+.
</ParamField>

<Note>
  The `token_type` and `policy` fields are deprecated and have been removed in Consul 1.11. Use `consul_policies`, `consul_roles`, `service_identities`, and `node_identities` instead.
</Note>

## Generating tokens

```bash
vault read consul/creds/my-role
```

Example output:

```
Key                Value
---                -----
lease_id           consul/creds/my-role/a2e2b514-f5c2-7823-6b0a-3d30f832c24e
lease_duration     1h
lease_renewable    true
access_or_secret   d782d08a-6b6a-45c4-a7d6-5e8d5de79b64
local              false
token              3e1bfe01-e9c0-fa39-4b3b-a3b62a8efbf3
```

Via the API:

```bash
curl \
  --header "X-Vault-Token: $VAULT_TOKEN" \
  $VAULT_ADDR/v1/consul/creds/my-role
```

## Token metadata

Vault sets the token's description to identify it as Vault-managed:

```
Vault <mount_accessor> Token <display_name>
```

This makes it straightforward to identify Vault-generated tokens in Consul's token list.

## Listing roles

```bash
vault list consul/roles
```

## Deleting a role

```bash
vault delete consul/roles/my-role
```

## Example: Application service mesh access

A common pattern is creating a role per service that grants that service the minimal Consul permissions it needs:

```bash
# First, create the policy in Consul
consul acl policy create \
  -name web-service \
  -description "Allow web service to discover backends" \
  -rules @web-service-policy.hcl

# Create the Vault role referencing that policy
vault write consul/roles/web-service \
  consul_policies=web-service \
  ttl=1h

# The application reads credentials at startup
vault read consul/creds/web-service
```

When the lease expires, Vault automatically deletes the token from Consul, ensuring no orphaned tokens accumulate over time.
