---
title: "TOTP Secrets Engine"
description: "The TOTP secrets engine manages TOTP keys and generates or validates time-based one-time password codes."
---

The TOTP (Time-Based One-Time Password) secrets engine generates and validates TOTP codes based on [RFC 6238](https://datatracker.ietf.org/doc/html/rfc6238). Vault can act as both a TOTP code generator (like an authenticator app) and as a TOTP code validator for multi-factor authentication flows.

## Enabling the engine

```bash
vault secrets enable totp
```

## Creating a TOTP key

A TOTP key stores the shared secret and configuration used to generate codes. Keys can either be generated by Vault or imported from an external source.

### Vault-generated key

When `generate=true`, Vault creates a new random key and returns a provisioning URL and QR code for enrollment in an authenticator app:

```bash
vault write totp/keys/my-service \
  generate=true \
  issuer="MyApp" \
  account_name="user@example.com"
```

Response:

```
Key       Value
---       -----
barcode   iVBORw0KGgoAAAANSUhEUgAAAMgAAADICAYAAACtWK6eAAA...
url       otpauth://totp/MyApp:user@example.com?algorithm=SHA1&digits=6&issuer=MyApp&period=30&secret=JBSWY3DPEHPK3PXP
```

Provide the `url` or `barcode` to the user to scan in their TOTP authenticator app (Google Authenticator, Authy, 1Password, etc.).

To suppress the QR code and only return the URL:

```bash
vault write totp/keys/my-service \
  generate=true \
  issuer="MyApp" \
  account_name="user@example.com" \
  qr_size=0
```

To generate the key without returning any enrollment data (store internally only):

```bash
vault write totp/keys/my-service \
  generate=true \
  issuer="MyApp" \
  account_name="user@example.com" \
  exported=false
```

### Importing an existing key

To import a key from an existing TOTP provisioning URL:

```bash
vault write totp/keys/my-service \
  url="otpauth://totp/MyApp:user@example.com?secret=JBSWY3DPEHPK3PXP&issuer=MyApp"
```

Or provide the raw shared secret:

```bash
vault write totp/keys/my-service \
  key=JBSWY3DPEHPK3PXP
```

### Key parameters

<ParamField path="generate" type="boolean" default="false">
  Whether Vault should generate the key. If `false`, a `key` or `url` must be provided.
</ParamField>

<ParamField path="exported" type="boolean" default="true">
  If `generate=true`, whether to return the QR code and URL in the response.
</ParamField>

<ParamField path="key_size" type="number" default="20">
  Size in bytes of the generated key. Only used when `generate=true`.
</ParamField>

<ParamField path="issuer" type="string">
  Name of the issuing organization. Required if `generate=true`. Appears in authenticator app as the account issuer.
</ParamField>

<ParamField path="account_name" type="string">
  Name of the account. Required if `generate=true`. Appears in authenticator app as the account name.
</ParamField>

<ParamField path="period" type="number" default="30">
  Time step in seconds for code generation (must match the authenticator app configuration).
</ParamField>

<ParamField path="algorithm" type="string" default="SHA1">
  HMAC algorithm for code generation. Options: `SHA1`, `SHA256`, `SHA512`.
</ParamField>

<ParamField path="digits" type="number" default="6">
  Number of digits in the generated code. Must be `6` or `8`.
</ParamField>

<ParamField path="skew" type="number" default="1">
  Number of time-step delay periods allowed during validation. `0` or `1`.
</ParamField>

<ParamField path="qr_size" type="number" default="200">
  Pixel size of the generated QR code square. Set to `0` to skip QR code generation.
</ParamField>

## Generating codes

Read the current TOTP code for a key:

```bash
vault read totp/code/my-service
```

Example output:

```
Key     Value
---     -----
code    623841
```

Via the API:

```bash
curl \
  --header "X-Vault-Token: $VAULT_TOKEN" \
  $VAULT_ADDR/v1/totp/code/my-service
```

<Note>
  Vault generates the code using the current system time and the stored key. The code changes every `period` seconds (default 30). The `skew` parameter allows validation to accept codes from adjacent time windows.
</Note>

## Validating codes

Validate a code provided by a user against a stored key:

```bash
vault write totp/code/my-service code=623841
```

Response:

```
Key      Value
---      -----
valid    true
```

Via the API:

```bash
curl \
  --header "X-Vault-Token: $VAULT_TOKEN" \
  --request POST \
  --data '{"code":"623841"}' \
  $VAULT_ADDR/v1/totp/code/my-service
```

## Listing keys

```bash
vault list totp/keys
```

## Deleting a key

```bash
vault delete totp/keys/my-service
```

## Use cases

### 2FA for internal services

Services can use Vault's TOTP engine as their TOTP backend, offloading key storage and validation to Vault.

1. At user enrollment, generate a key with `generate=true, exported=true`
2. Return the `url` or `barcode` to the user to enroll in their authenticator app
3. At login, call `vault write totp/code/<key-name> code=<user-provided-code>` and check the `valid` field

### MFA backup codes service

Store TOTP keys for multiple users:

```bash
# Create a key per user
vault write totp/keys/user-alice@example.com \
  generate=true \
  issuer="CorpApp" \
  account_name="alice@example.com"

vault write totp/keys/user-bob@example.com \
  generate=true \
  issuer="CorpApp" \
  account_name="bob@example.com"
```

Use Vault policies to restrict each service to only reading or validating its own users' keys.

### Service account MFA

For non-interactive service accounts that need TOTP validation:

1. Import the shared TOTP secret (obtained out-of-band from the service provider)
2. Use `vault read totp/code/<key>` in automation to get the current code
3. Include the code in the service's authentication request to the external provider
