---
title: "Database Secrets Engine"
description: "The Database secrets engine generates dynamic credentials for relational and NoSQL databases with automatic revocation."
---

The Database secrets engine generates dynamic usernames and passwords for databases on demand. When the lease expires, Vault automatically revokes the credentials. This eliminates long-lived database passwords and reduces the blast radius of a credential compromise.

## Supported databases

Vault's database engine uses a plugin architecture. Built-in plugins include:

| Database | Plugin name |
|---|---|
| PostgreSQL | `postgresql-database-plugin` |
| MySQL / MariaDB | `mysql-database-plugin` |
| MySQL (RDS) | `mysql-rds-database-plugin` |
| MySQL (Aurora) | `mysql-aurora-database-plugin` |
| MongoDB | `mongodb-database-plugin` |
| MongoDB Atlas | `mongodbatlas-database-plugin` |
| Cassandra | `cassandra-database-plugin` |
| Elasticsearch | `elasticsearch-database-plugin` |
| Snowflake | `snowflake-database-plugin` |
| Redis | `redis-database-plugin` |
| MSSQL | `mssql-database-plugin` |

## Enabling the engine

```bash
vault secrets enable database
```

## Configuring a database connection

<Steps>
  <Step title="Enable the engine">
    ```bash
    vault secrets enable database
    ```
  </Step>

  <Step title="Configure the connection">
    For PostgreSQL:

    ```bash
    vault write database/config/my-postgres \
      plugin_name=postgresql-database-plugin \
      allowed_roles="my-role,readonly" \
      connection_url="postgresql://{{username}}:{{password}}@localhost:5432/mydb?sslmode=disable" \
      username=vaultuser \
      password=vaultpassword
    ```

    For MySQL:

    ```bash
    vault write database/config/my-mysql \
      plugin_name=mysql-database-plugin \
      allowed_roles="my-role" \
      connection_url="{{username}}:{{password}}@tcp(127.0.0.1:3306)/" \
      username=vaultuser \
      password=vaultpassword
    ```

    For MongoDB:

    ```bash
    vault write database/config/my-mongo \
      plugin_name=mongodb-database-plugin \
      allowed_roles="my-role" \
      connection_url="mongodb://{{username}}:{{password}}@localhost:27017/admin" \
      username=vaultuser \
      password=vaultpassword
    ```

    <Note>
      The `{{username}}` and `{{password}}` template variables are substituted by Vault at connection time.
    </Note>
  </Step>

  <Step title="Rotate root credentials">
    After configuring a connection, immediately rotate the root credentials so that only Vault knows the password:

    ```bash
    vault write -force database/rotate-root/my-postgres
    ```

    <Warning>
      After rotating root credentials, the original password is no longer valid. Ensure Vault is correctly configured before rotating.
    </Warning>
  </Step>
</Steps>

### Connection parameters

<ParamField path="plugin_name" type="string" required>
  The name of the database plugin.
</ParamField>

<ParamField path="connection_url" type="string" required>
  Connection string template. Use `{{username}}` and `{{password}}` for the credentials Vault manages.
</ParamField>

<ParamField path="allowed_roles" type="string">
  Comma-separated list of roles allowed to use this connection. Use `*` to allow all roles.
</ParamField>

<ParamField path="verify_connection" type="boolean" default="true">
  Whether to verify the connection when configuring the plugin.
</ParamField>

<ParamField path="password_policy" type="string">
  Name of the password policy to use for generated passwords.
</ParamField>

## Creating a dynamic role

Roles define the SQL statements used to create and revoke database users.

```bash
vault write database/roles/my-role \
  db_name=my-postgres \
  creation_statements="CREATE ROLE \"{{name}}\" WITH LOGIN PASSWORD '{{password}}' VALID UNTIL '{{expiration}}'; GRANT ALL PRIVILEGES ON ALL TABLES IN SCHEMA public TO \"{{name}}\";" \
  revocation_statements="REVOKE ALL PRIVILEGES ON ALL TABLES IN SCHEMA public FROM \"{{name}}\"; REVOKE ALL PRIVILEGES ON ALL SEQUENCES IN SCHEMA public FROM \"{{name}}\"; DROP ROLE IF EXISTS \"{{name}}\";" \
  default_ttl=1h \
  max_ttl=24h
```

For MySQL:

```bash
vault write database/roles/my-role \
  db_name=my-mysql \
  creation_statements="CREATE USER '{{name}}'@'%' IDENTIFIED BY '{{password}}'; GRANT SELECT ON mydb.* TO '{{name}}'@'%';" \
  default_ttl=1h \
  max_ttl=24h
```

### Dynamic role parameters

<ParamField path="db_name" type="string" required>
  The name of the database connection to use (as configured at `database/config/<name>`).
</ParamField>

<ParamField path="creation_statements" type="string[]">
  Database statements executed to create and configure the user. Available template variables: `{{name}}`, `{{password}}`, `{{expiration}}`.
</ParamField>

<ParamField path="revocation_statements" type="string[]">
  Database statements executed to revoke the user when the lease expires.
</ParamField>

<ParamField path="rollback_statements" type="string[]">
  Statements executed to roll back a failed creation. Not supported by all plugins.
</ParamField>

<ParamField path="renew_statements" type="string[]">
  Statements executed to extend a user's expiration time on renewal.
</ParamField>

<ParamField path="default_ttl" type="string">
  Default TTL for generated credentials.
</ParamField>

<ParamField path="max_ttl" type="string">
  Maximum TTL for generated credentials.
</ParamField>

<ParamField path="credential_type" type="string" default="password">
  Type of credential to generate. Options: `password`, `rsa_private_key`, `client_certificate`.
</ParamField>

## Generating credentials

```bash
vault read database/creds/my-role
```

Example output:

```
Key                Value
---                -----
lease_id           database/creds/my-role/3e0b1522-37e4-b0e4-e3f0-23a2f4a2d6c8
lease_duration     1h
lease_renewable    true
password           A1a-xzy789
username           v-token-my-role-xzy789-1609459200
```

Via the API:

```bash
curl \
  --header "X-Vault-Token: $VAULT_TOKEN" \
  $VAULT_ADDR/v1/database/creds/my-role
```

## Static roles (credential rotation)

Static roles manage an existing database account and automatically rotate its password on a schedule. Unlike dynamic roles, static roles do not create or delete database users.

```bash
vault write database/static-roles/my-static-role \
  db_name=my-postgres \
  username=app_user \
  rotation_period=24h
```

With a cron-style rotation schedule:

```bash
vault write database/static-roles/nightly-rotation \
  db_name=my-postgres \
  username=reporting_user \
  rotation_schedule="0 2 * * *" \
  rotation_window=1h
```

### Static role parameters

<ParamField path="username" type="string" required>
  The existing database username Vault will manage. Cannot be changed after the role is created.
</ParamField>

<ParamField path="rotation_period" type="string">
  Duration between password rotations (e.g., `24h`). Mutually exclusive with `rotation_schedule`.
</ParamField>

<ParamField path="rotation_schedule" type="string">
  Cron-style schedule for rotation (e.g., `0 2 * * *` for 2 AM daily). Mutually exclusive with `rotation_period`.
</ParamField>

<ParamField path="rotation_window" type="string">
  Window of time in which rotation is allowed to occur, starting from the scheduled rotation time. Only valid with `rotation_schedule`.
</ParamField>

<ParamField path="rotation_statements" type="string[]">
  Database statements executed to rotate the user's credentials. If not provided, the plugin uses its default rotation statement.
</ParamField>

Read the current password for a static role:

```bash
vault read database/static-creds/my-static-role
```

Manually trigger a rotation:

```bash
vault write -force database/rotate-role/my-static-role
```

## Root credential rotation

After initial configuration, rotate the root credentials so Vault has exclusive knowledge of the database root password:

```bash
vault write -force database/rotate-root/my-postgres
```

Vault generates a new password, updates the database, and stores the new credentials internally.
