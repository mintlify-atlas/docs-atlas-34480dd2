---
title: "Transit Secrets Engine"
description: "The Transit secrets engine provides encryption-as-a-service, allowing applications to encrypt data without storing it in Vault."
---

The Transit secrets engine handles cryptographic functions on data in transit. It does not store the data sent to it — instead, it returns the result of the cryptographic operation. Think of it as a cryptography API.

Transit provides:
- Symmetric encryption and decryption (AES-GCM, ChaCha20-Poly1305)
- Asymmetric signing and verification (ECDSA, Ed25519, RSA)
- HMAC generation and verification
- Data key generation
- Key rotation and versioning
- Bring Your Own Key (BYOK) via key import

## Enabling the engine

```bash
vault secrets enable transit
```

## Key types

The Transit engine supports a wide range of key types, sourced from `builtin/logical/transit/path_keys.go`:

| Key Type | Category | Use case |
|---|---|---|
| `aes128-gcm96` | Symmetric | Encryption / decryption |
| `aes256-gcm96` | Symmetric | Encryption / decryption (default) |
| `chacha20-poly1305` | Symmetric | Encryption / decryption |
| `ecdsa-p256` | Asymmetric | Signing / verification |
| `ecdsa-p384` | Asymmetric | Signing / verification |
| `ecdsa-p521` | Asymmetric | Signing / verification |
| `ed25519` | Asymmetric | Signing / verification |
| `rsa-2048` | Asymmetric | Signing / encryption |
| `rsa-3072` | Asymmetric | Signing / encryption |
| `rsa-4096` | Asymmetric | Signing / encryption |
| `hmac` | MAC | HMAC generation |

## Creating a key

```bash
# Create an AES-256-GCM96 key (default)
vault write -f transit/keys/my-key

# Create a specific key type
vault write transit/keys/my-signing-key type=ed25519

# Create an exportable key
vault write transit/keys/my-key \
  type=aes256-gcm96 \
  exportable=true

# Create a key with automatic rotation
vault write transit/keys/my-key \
  type=aes256-gcm96 \
  auto_rotate_period=720h
```

### Key creation parameters

<ParamField path="type" type="string" default="aes256-gcm96">
  Key type. See the table above for all valid values.
</ParamField>

<ParamField path="derived" type="boolean" default="false">
  Enables key derivation mode, allowing per-transaction unique keys for encryption operations.
</ParamField>

<ParamField path="convergent_encryption" type="boolean" default="false">
  Enables convergent encryption (same plaintext + context = same ciphertext). Requires `derived=true`. Use with care — nonces must be unique per context.
</ParamField>

<ParamField path="exportable" type="boolean" default="false">
  Allows the raw key material to be exported via `transit/export`.
</ParamField>

<ParamField path="allow_plaintext_backup" type="boolean" default="false">
  Enables plaintext backup of all key versions via `transit/backup`. Once set, cannot be disabled.
</ParamField>

<ParamField path="auto_rotate_period" type="string" default="0">
  Duration between automatic key rotations (e.g., `720h`). `0` disables automatic rotation.
</ParamField>

## Encrypting data

Plaintext must be base64-encoded before sending to Vault.

```bash
# Encrypt a value
vault write transit/encrypt/my-key \
  plaintext=$(echo -n "my secret data" | base64)
```

Response:

```
Key           Value
---           -----
ciphertext    vault:v1:8SDd3WHDOjf7mq69CyCqYjBXAiQQAVZRkFM13ok481zoCmHnSeDX9vyf7w==
key_version   1
```

The `vault:v1:` prefix indicates the key version used. After a key rotation, new ciphertexts use `vault:v2:`, but old `vault:v1:` ciphertexts remain decryptable.

Via the API:

```bash
curl \
  --header "X-Vault-Token: $VAULT_TOKEN" \
  --request POST \
  --data "{\"plaintext\": \"$(echo -n 'my secret data' | base64)\"}" \
  $VAULT_ADDR/v1/transit/encrypt/my-key
```

## Decrypting data

```bash
vault write transit/decrypt/my-key \
  ciphertext="vault:v1:8SDd3WHDOjf7mq69CyCqYjBXAiQQAVZRkFM13ok481zoCmHnSeDX9vyf7w=="
```

Response:

```
Key          Value
---          -----
plaintext    bXkgc2VjcmV0IGRhdGE=
```

Decode the base64 result:

```bash
echo "bXkgc2VjcmV0IGRhdGE=" | base64 --decode
# my secret data
```

## Key rotation

Rotating a key creates a new key version. Old ciphertexts remain decryptable using the previous key version, but new encryptions use the latest version.

```bash
# Rotate the key
vault write -f transit/keys/my-key/rotate

# Read key info to see all versions
vault read transit/keys/my-key
```

After rotation, upgrade existing ciphertexts to the new key version with `rewrap`:

```bash
vault write transit/rewrap/my-key \
  ciphertext="vault:v1:8SDd3WHDOjf7mq69CyCqYjBXAiQQAVZRkFM13ok481zoCmHnSeDX9vyf7w=="
```

This re-encrypts the ciphertext with the latest key version without exposing the plaintext.

### Key version management

```bash
vault write transit/keys/my-key/config \
  min_decryption_version=2 \
  min_encryption_version=3
```

<ParamField path="min_decryption_version" type="number" default="1">
  The minimum key version that can be used for decryption. Setting this to a higher value effectively invalidates older ciphertexts.
</ParamField>

<ParamField path="min_encryption_version" type="number" default="0">
  The minimum key version used for encryption. `0` means always use the latest version.
</ParamField>

## HMAC and signing

### HMAC

```bash
# Generate an HMAC
vault write transit/hmac/my-key \
  input=$(echo -n "data to sign" | base64)

# Verify an HMAC
vault write transit/verify/my-key \
  input=$(echo -n "data to sign" | base64) \
  hmac="vault:v1:QL9C..."
```

### Signing (asymmetric keys)

```bash
# Create an Ed25519 key
vault write -f transit/keys/my-signing-key type=ed25519

# Sign data
vault write transit/sign/my-signing-key \
  input=$(echo -n "data to sign" | base64)

# Verify a signature
vault write transit/verify/my-signing-key \
  input=$(echo -n "data to sign" | base64) \
  signature="vault:v1:MEUCIQDy..."
```

## Bring Your Own Key (BYOK)

Import an existing key into Transit using the wrapping key mechanism.

```bash
# Get the wrapping key
vault read transit/wrapping_key

# Wrap your key material with the wrapping key, then import
vault write transit/keys/my-imported-key/import \
  ciphertext="<wrapped_key_material>" \
  type=aes256-gcm96
```

<Note>
  Imported keys can be rotated within Vault to generate new Vault-native key versions, but the original imported version remains as version 1.
</Note>

## Data key generation

Generate a data key — a random key encrypted under a Transit key. The application uses the plaintext key locally for encryption and stores only the ciphertext.

```bash
# Returns both plaintext and ciphertext versions of a new random key
vault write transit/datakey/plaintext/my-key plaintext=""

# Returns only the ciphertext version (more secure — plaintext never leaves Vault)
vault write transit/datakey/wrapped/my-key plaintext=""
```

The application:
1. Requests a data key from Vault
2. Encrypts local data using the plaintext key
3. Stores the ciphertext key alongside the encrypted data
4. Discards the plaintext key from memory
5. Later, sends the ciphertext key back to Vault to decrypt and retrieve the plaintext key for decryption

## Batch operations

Transit supports batch encrypt and decrypt requests to reduce API call overhead:

```bash
curl \
  --header "X-Vault-Token: $VAULT_TOKEN" \
  --request POST \
  --data '{
    "batch_input": [
      {"plaintext": "dGVzdA=="},
      {"plaintext": "aGVsbG8="}
    ]
  }' \
  $VAULT_ADDR/v1/transit/encrypt/my-key
```

## Key deletion

By default, Transit keys cannot be deleted. Enable deletion explicitly:

```bash
vault write transit/keys/my-key/config deletion_allowed=true
vault delete transit/keys/my-key
```
