---
title: "PKI Secrets Engine"
description: "The PKI secrets engine generates X.509 certificates and acts as a certificate authority for internal services."
---

The PKI secrets engine generates dynamic X.509 certificates. It acts as a certificate authority (CA), issuing certificates with configurable subject fields, key types, TTLs, and SANs. Unlike static certificates, Vault-issued certificates have short TTLs and are revoked automatically when their lease expires.

## Key concepts

- **Root CA**: Self-signed certificate authority. Store offline or in Vault's PKI engine.
- **Intermediate CA**: Signed by the root CA. Vault typically operates as an intermediate CA in production.
- **Roles**: Define the parameters for certificate issuance (allowed domains, key type, TTL, etc.).
- **Issuers**: Named CA certificates within a single PKI mount. Vault supports multiple issuers per mount.

## Setup

### Root CA

<Steps>
  <Step title="Enable the PKI engine">
    ```bash
    vault secrets enable pki
    ```
  </Step>

  <Step title="Configure the maximum TTL">
    ```bash
    vault secrets tune -max-lease-ttl=87600h pki
    ```
  </Step>

  <Step title="Generate a root CA">
    ```bash
    vault write pki/root/generate/internal \
      common_name="example.com" \
      issuer_name="root-2024" \
      ttl=87600h
    ```

    For an **exported** root (Vault returns the private key — use with caution):

    ```bash
    vault write pki/root/generate/exported \
      common_name="example.com" \
      ttl=87600h
    ```

    <Warning>
      In production, generate the root CA outside Vault and import it, or use an offline root and have Vault operate only as an intermediate CA.
    </Warning>
  </Step>

  <Step title="Configure CRL and issuing certificate URLs">
    ```bash
    vault write pki/config/urls \
      issuing_certificates="$VAULT_ADDR/v1/pki/ca" \
      crl_distribution_points="$VAULT_ADDR/v1/pki/crl"
    ```
  </Step>
</Steps>

### Intermediate CA

<Steps>
  <Step title="Enable a second PKI mount for the intermediate CA">
    ```bash
    vault secrets enable -path=pki_int pki
    vault secrets tune -max-lease-ttl=43800h pki_int
    ```
  </Step>

  <Step title="Generate an intermediate CSR">
    ```bash
    vault write -format=json pki_int/intermediate/generate/internal \
      common_name="example.com Intermediate Authority" \
      issuer_name="example-dot-com-intermediate" \
      | jq -r '.data.csr' > pki_intermediate.csr
    ```
  </Step>

  <Step title="Sign the intermediate CSR with the root CA">
    ```bash
    vault write -format=json pki/root/sign-intermediate \
      issuer_ref="root-2024" \
      csr=@pki_intermediate.csr \
      format=pem_bundle \
      ttl=43800h \
      | jq -r '.data.certificate' > intermediate.cert.pem
    ```
  </Step>

  <Step title="Import the signed certificate into the intermediate mount">
    ```bash
    vault write pki_int/intermediate/set-signed \
      certificate=@intermediate.cert.pem
    ```
  </Step>
</Steps>

## Creating a role

Roles define the parameters for certificates issued by this PKI mount.

```bash
vault write pki_int/roles/example-dot-com \
  issuer_ref="example-dot-com-intermediate" \
  allowed_domains="example.com" \
  allow_subdomains=true \
  max_ttl=720h
```

### Role parameters

<ParamField path="allowed_domains" type="string">
  Comma-separated list of domains this role can issue certificates for. Used with `allow_bare_domains`, `allow_subdomains`, and `allow_glob_domains`.
</ParamField>

<ParamField path="allow_subdomains" type="boolean" default="false">
  If set, clients can request certificates for subdomains of `allowed_domains`, including wildcard subdomains.
</ParamField>

<ParamField path="allow_bare_domains" type="boolean" default="false">
  If set, clients can request certificates for the exact domains in `allowed_domains` (e.g., `example.com` itself).
</ParamField>

<ParamField path="allow_wildcard_certificates" type="boolean" default="true">
  If set, wildcard certificates (e.g., `*.example.com`) are allowed.
</ParamField>

<ParamField path="allow_any_name" type="boolean" default="false">
  If set, clients can request certificates for any domain, bypassing `allowed_domains` restrictions. Use with caution.
</ParamField>

<ParamField path="allow_ip_sans" type="boolean" default="true">
  If set, IP Subject Alternative Names are allowed.
</ParamField>

<ParamField path="key_type" type="string" default="rsa">
  Key type for generated certificates. Valid values: `rsa`, `ec`, `ed25519`, `any`.
</ParamField>

<ParamField path="key_bits" type="number" default="0">
  Key size. For RSA: 2048, 3072, or 4096. For EC: 224, 256, 384, or 521. `0` uses the type default.
</ParamField>

<ParamField path="ttl" type="string">
  Default certificate TTL if no specific TTL is requested.
</ParamField>

<ParamField path="max_ttl" type="string">
  Maximum allowed certificate TTL.
</ParamField>

<ParamField path="no_store" type="boolean" default="false">
  If set, issued certificates are not stored in Vault. They cannot be enumerated or revoked. Use only for short-lived, non-sensitive certificates.
</ParamField>

<ParamField path="generate_lease" type="boolean" default="false">
  If set, issued certificates will have Vault leases. This allows revocation via `vault revoke <lease_id>` but is not recommended for large certificate volumes.
</ParamField>

<ParamField path="issuer_ref" type="string" default="default">
  Reference to the issuer (named CA certificate) that signs requests for this role.
</ParamField>

## Issuing certificates

```bash
vault write pki_int/issue/example-dot-com \
  common_name="test.example.com" \
  ttl=24h
```

The response contains:
- `certificate`: The issued certificate (PEM)
- `issuing_ca`: The issuing CA certificate (PEM)
- `ca_chain`: Full CA chain (PEM)
- `private_key`: The private key (PEM, only on initial issuance)
- `serial_number`: Certificate serial number

Via the API:

```bash
curl \
  --header "X-Vault-Token: $VAULT_TOKEN" \
  --request POST \
  --data '{"common_name":"test.example.com","ttl":"24h"}' \
  $VAULT_ADDR/v1/pki_int/issue/example-dot-com
```

With SANs:

```bash
vault write pki_int/issue/example-dot-com \
  common_name="app.example.com" \
  alt_names="app-internal.example.com,app.corp.example.com" \
  ip_sans="10.0.0.1" \
  ttl=72h
```

## CRL configuration

Vault automatically maintains a Certificate Revocation List (CRL).

```bash
# Configure CRL settings
vault write pki/config/crl \
  expiry=72h \
  disable=false

# Rotate (rebuild) the CRL immediately
vault write pki/crl/rotate
```

The CRL is available at `$VAULT_ADDR/v1/pki/crl` and can be fetched in PEM or DER format.

## Certificate revocation

```bash
# Revoke a certificate by serial number
vault write pki/revoke \
  serial_number="39:dd:2e:90:b7:23:1f:8d:d3:7d:31:c5:1b:da:84:d0:5b:65:31:58"
```

After revocation, the serial number appears in the CRL at the next CRL rotation.

## Auto-rotation with Vault Agent

Vault Agent can automatically renew or re-issue certificates before they expire. Configure a `template` stanza pointing to the PKI issue endpoint:

```hcl
template {
  source      = "/etc/vault-templates/cert.tpl"
  destination = "/etc/ssl/certs/app.pem"
  error_on_missing_key = true
}
```

In the template:

```hcl
{{ with pkiCert "pki_int/issue/example-dot-com" "common_name=app.example.com" "ttl=24h" }}
{{ .Cert }}{{ .CA }}{{ .Key }}
{{ end }}
```

Vault Agent will re-issue the certificate when it reaches 2/3 of its TTL by default.

## Tidy

Over time, Vault accumulates expired certificates in storage. Use `pki/tidy` to clean them up:

```bash
vault write pki/tidy \
  tidy_cert_store=true \
  tidy_revoked_certs=true \
  safety_buffer=72h
```
