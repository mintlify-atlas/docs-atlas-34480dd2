title: "Secrets Engines"
description: "Secrets engines are plugins that handle the storage, generation, or encryption of secrets in Vault."
---

Secrets engines are components in HashiCorp Vault that store, generate, or encrypt data. They receive API calls and decide what to do with them. Some engines simply store and read data (like a KV store), while others connect to external services and generate dynamic credentials on demand.

## What are secrets engines?

A secrets engine is mounted at a specific path in Vault. When a request comes to Vault for a path, the router sends the request to whichever secrets engine is mounted at that path. The engine handles the request and produces a response.

Vault ships with many built-in secrets engines, and can also load external plugins.

### Static vs. dynamic secrets

<Tabs>
  <Tab title="Static secrets">
    Static secrets are fixed values stored in Vault. When you read a static secret, you always get the same value back (unless someone writes a new value).

    The **KV secrets engine** is the primary example. You write a secret once and retrieve it as needed.

    ```bash
    vault kv put secret/myapp username=admin password=s3cr3t
    vault kv get secret/myapp
    ```
  </Tab>
  <Tab title="Dynamic secrets">
    Dynamic secrets are generated on demand. Vault creates a credential when requested, attaches a lease to it, and automatically revokes it when the lease expires.

    The **AWS**, **Database**, **SSH**, and **Consul** engines all support dynamic secret generation.

    ```bash
    vault read aws/creds/my-role
    # Returns a temporary AWS access key and secret that expire automatically
    ```

    Dynamic secrets reduce the risk of credential exposure because each consumer gets a unique, short-lived credential.
  </Tab>
</Tabs>

## Managing secrets engines

<Steps>
  <Step title="Enable a secrets engine">
    Use `vault secrets enable` to mount a secrets engine at a path.

    ```bash
    # Enable KV v2 at the default 'secret/' path
    vault secrets enable -path=secret kv-v2

    # Enable the AWS secrets engine
    vault secrets enable aws

    # Enable the database secrets engine at a custom path
    vault secrets enable -path=db database
    ```

    <Note>
      Each secrets engine instance is isolated. You can mount the same engine type at multiple paths with different configurations.
    </Note>
  </Step>

  <Step title="List enabled secrets engines">
    ```bash
    vault secrets list
    ```

    Example output:

    ```
    Path          Type         Accessor              Description
    ----          ----         --------              -----------
    aws/          aws          aws_abc123            n/a
    cubbyhole/    cubbyhole    cubbyhole_xyz789      per-token private secret storage
    database/     database     database_def456       n/a
    identity/     identity     identity_ghi012       identity store
    secret/       kv           kv_jkl345             key/value secret storage
    sys/          system       system_mno678         system endpoints used for control, policy and debugging
    ```
  </Step>

  <Step title="Configure the secrets engine">
    After enabling an engine, configure it with connection credentials and options. The configuration path varies by engine type.

    ```bash
    # Configure AWS root credentials
    vault write aws/config/root \
      access_key=AKIAIOSFODNN7EXAMPLE \
      secret_key=wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY \
      region=us-east-1

    # Configure a database connection
    vault write database/config/my-postgres \
      plugin_name=postgresql-database-plugin \
      connection_url="postgresql://{{username}}:{{password}}@localhost:5432/mydb"
    ```
  </Step>

  <Step title="Create roles">
    Most secrets engines use roles to define what credentials to create and with what permissions.

    ```bash
    vault write aws/roles/my-role \
      credential_type=iam_user \
      policy_arns=arn:aws:iam::aws:policy/ReadOnlyAccess
    ```
  </Step>

  <Step title="Request credentials">
    Read from the engine's credential path to generate secrets.

    ```bash
    vault read aws/creds/my-role
    vault read database/creds/my-role
    vault read consul/creds/my-role
    ```
  </Step>

  <Step title="Disable a secrets engine">
    <Warning>
      Disabling a secrets engine revokes all active secrets and leases generated by that engine. This is irreversible.
    </Warning>

    ```bash
    vault secrets disable secret/
    ```
  </Step>
</Steps>

## Leases and renewal

When Vault generates a dynamic secret, it attaches a **lease** to that secret. A lease is metadata containing:

- A lease ID
- A TTL (time-to-live)
- A renewable flag

Applications can renew leases before they expire to keep using the credential. When a lease expires or is revoked, Vault calls the secrets engine's revocation logic to delete the credential from the external system.

```bash
# Renew a lease
vault lease renew <lease_id>

# Revoke a lease immediately
vault lease revoke <lease_id>
```

## API

All secrets engines expose a REST API under their mount path. You can interact with them directly via `curl`:

```bash
curl \
  --header "X-Vault-Token: $VAULT_TOKEN" \
  --request GET \
  $VAULT_ADDR/v1/secret/data/myapp
```
