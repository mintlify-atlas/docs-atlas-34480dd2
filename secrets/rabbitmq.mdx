---
title: "RabbitMQ Secrets Engine"
description: "The RabbitMQ secrets engine generates dynamic credentials for RabbitMQ with configurable vhost permissions."
---

The RabbitMQ secrets engine generates dynamic RabbitMQ user credentials on demand. Vault creates a new RabbitMQ user for each credential request, assigns the configured virtual host permissions and tags, and revokes the user when the lease expires.

## Enabling the engine

```bash
vault secrets enable rabbitmq
```

## Configuring the RabbitMQ connection

Vault needs a RabbitMQ administrator credential to create and delete users.

```bash
vault write rabbitmq/config/connection \
  connection_uri="http://localhost:15672" \
  username=admin \
  password=adminpassword
```

### Connection parameters

<ParamField path="connection_uri" type="string" required>
  The RabbitMQ management API URI. Typically `http://<host>:15672` or `https://<host>:15671` for TLS.
</ParamField>

<ParamField path="username" type="string" required>
  Administrator username for the RabbitMQ management API.
</ParamField>

<ParamField path="password" type="string" required>
  Administrator password for the RabbitMQ management API.
</ParamField>

<ParamField path="username_template" type="string">
  Go template for generated usernames. Defaults to `{{ printf "%s-%s" (.DisplayName) (uuid) }}`.
</ParamField>

<ParamField path="password_policy" type="string">
  Name of a Vault password policy to use for generating passwords.
</ParamField>

## Configuring the lease TTL

Configure the default and maximum lease durations for generated credentials:

```bash
vault write rabbitmq/config/lease \
  ttl=1h \
  max_ttl=24h
```

<ParamField path="ttl" type="string">
  Default TTL for generated credentials.
</ParamField>

<ParamField path="max_ttl" type="string">
  Maximum TTL for generated credentials.
</ParamField>

## Creating roles

Roles define the tags and virtual host permissions granted to generated users. The role structure is defined in `builtin/logical/rabbitmq/path_roles.go`.

### Role with tags only

```bash
vault write rabbitmq/roles/monitoring \
  tags=monitoring
```

### Role with vhost permissions

```bash
vault write rabbitmq/roles/app-producer \
  vhosts='{"/": {"configure": "", "write": ".*", "read": ""}}'
```

### Role with full permissions on multiple vhosts

```bash
vault write rabbitmq/roles/admin-role \
  tags=administrator \
  vhosts='{"/": {"configure": ".*", "write": ".*", "read": ".*"}, "/app": {"configure": ".*", "write": ".*", "read": ".*"}}'
```

### Role with topic permissions

```bash
vault write rabbitmq/roles/topic-role \
  vhosts='{"/": {"configure": "", "write": ".*", "read": ""}}' \
  vhost_topics='{"/": {"amq.topic": {"write": ".*", "read": ""}}}'  
```

### Role parameters

<ParamField path="tags" type="string">
  Comma-separated list of RabbitMQ user tags. Common values: `administrator`, `monitoring`, `management`, `policymaker`, `impersonator`.
</ParamField>

<ParamField path="vhosts" type="string">
  JSON string mapping virtual host names to permission objects. Each permission object has `configure`, `write`, and `read` fields accepting AMQP resource name regex patterns.

  ```json
  {
    "/": {
      "configure": ".*",
      "write": ".*",
      "read": ".*"
    }
  }
  ```
</ParamField>

<ParamField path="vhost_topics" type="string">
  JSON string mapping virtual host names to exchange names to topic permission objects. Each topic permission object has `write` and `read` fields.

  ```json
  {
    "/": {
      "amq.topic": {
        "write": "myqueue.*",
        "read": "myqueue.*"
      }
    }
  }
  ```
</ParamField>

<Note>
  Either `tags` or `vhosts` must be provided when creating a role. If neither is specified, Vault returns an error.
</Note>

## Generating credentials

```bash
vault read rabbitmq/creds/my-role
```

Example output:

```
Key                Value
---                -----
lease_id           rabbitmq/creds/my-role/8e8b5a86-e960-b3e4-b9e0-0f7e4a2d3e5b
lease_duration     1h
lease_renewable    true
password           A1b-xzy789ABCdef
username           root-a72b3c4d-e5f6-7890-abcd-ef1234567890
```

Via the API:

```bash
curl \
  --header "X-Vault-Token: $VAULT_TOKEN" \
  $VAULT_ADDR/v1/rabbitmq/creds/my-role
```

The generated username follows the template `{{ printf "%s-%s" (.DisplayName) (uuid) }}` by default, as seen in `builtin/logical/rabbitmq/path_role_create.go`.

## Lease revocation

When a lease expires or is revoked, Vault calls `DELETE /api/users/<username>` on the RabbitMQ management API to delete the user and all of its permissions.

```bash
# Revoke credentials immediately
vault lease revoke rabbitmq/creds/my-role/8e8b5a86-e960-b3e4-b9e0-0f7e4a2d3e5b

# Renew credentials before expiry
vault lease renew rabbitmq/creds/my-role/8e8b5a86-e960-b3e4-b9e0-0f7e4a2d3e5b
```

## Listing roles

```bash
vault list rabbitmq/roles
```

## Example: Producer and consumer roles

A typical pattern separates producer and consumer permissions:

```bash
# Producer: can write to queues, cannot read or configure
vault write rabbitmq/roles/producer \
  vhosts='{"/app": {"configure": "", "write": ".*", "read": ""}}'

# Consumer: can read from queues, cannot write or configure
vault write rabbitmq/roles/consumer \
  vhosts='{"/app": {"configure": "", "write": "", "read": ".*"}}'

# Generate producer credentials
vault read rabbitmq/creds/producer

# Generate consumer credentials
vault read rabbitmq/creds/consumer
```

Each service gets its own unique credential, scoped to only the operations it needs. When the lease expires, Vault deletes the RabbitMQ user automatically.
