---
title: "SSH Secrets Engine"
description: "The SSH secrets engine provides two modes of SSH authentication: OTP-based one-time passwords and CA-signed certificates."
---

The SSH secrets engine secures SSH access to machines by providing short-lived credentials. It supports two distinct modes, sourced from `builtin/logical/ssh/path_roles.go`:

- **CA mode** (recommended): Vault acts as an SSH certificate authority, signing user or host public keys
- **OTP mode**: Vault generates single-use passwords that are valid only once

## CA mode

CA mode is the recommended approach. Vault holds an SSH CA key pair. When a user wants to SSH to a host, they submit their public key to Vault, which signs it and returns a certificate. The host trusts the CA public key, so it accepts the signed certificate.

<Steps>
  <Step title="Enable the SSH engine">
    ```bash
    vault secrets enable -path=ssh-client-signer ssh
    ```
  </Step>

  <Step title="Generate the CA key pair">
    ```bash
    vault write ssh-client-signer/config/ca generate_signing_key=true
    ```

    To use an existing key pair:

    ```bash
    vault write ssh-client-signer/config/ca \
      private_key=@/path/to/ca_key \
      public_key=@/path/to/ca_key.pub
    ```
  </Step>

  <Step title="Configure trusted hosts">
    Distribute the CA public key to all SSH servers so they trust Vault-signed certificates:

    ```bash
    # Get the CA public key
    vault read -field=public_key ssh-client-signer/config/ca > /etc/ssh/vault_ca.pub
    ```

    Add to `/etc/ssh/sshd_config` on each host:

    ```bash
    TrustedUserCAKeys /etc/ssh/vault_ca.pub
    ```
  </Step>

  <Step title="Create a role">
    ```bash
    vault write ssh-client-signer/roles/my-role \
      key_type=ca \
      allowed_users="ubuntu,ec2-user,admin" \
      allow_user_certificates=true \
      default_extensions="permit-pty=,permit-port-forwarding=" \
      ttl=30m
    ```
  </Step>

  <Step title="Sign a public key">
    The user submits their SSH public key for signing:

    ```bash
    vault write ssh-client-signer/sign/my-role \
      public_key=@$HOME/.ssh/id_rsa.pub
    ```

    Save the certificate:

    ```bash
    vault write -field=signed_key \
      ssh-client-signer/sign/my-role \
      public_key=@$HOME/.ssh/id_rsa.pub \
      > $HOME/.ssh/id_rsa-cert.pub
    ```
  </Step>

  <Step title="Connect via SSH">
    ```bash
    ssh -i $HOME/.ssh/id_rsa-cert.pub -i $HOME/.ssh/id_rsa ubuntu@host
    ```
  </Step>
</Steps>

### CA role parameters

<ParamField path="key_type" type="string" required>
  Must be `ca` for CA mode.
</ParamField>

<ParamField path="allowed_users" type="string">
  Comma-separated list of usernames that can be specified as certificate principals. Use `*` to allow any user.
</ParamField>

<ParamField path="allowed_domains" type="string">
  If set, only the listed domains are valid for host certificate principals.
</ParamField>

<ParamField path="allow_user_certificates" type="boolean" default="false">
  If set, certificates are allowed to be signed for use as a user certificate.
</ParamField>

<ParamField path="allow_host_certificates" type="boolean" default="false">
  If set, certificates are allowed to be signed for use as a host certificate.
</ParamField>

<ParamField path="ttl" type="string">
  Default certificate validity period. Defaults to `max_ttl`.
</ParamField>

<ParamField path="max_ttl" type="string">
  Maximum allowed certificate validity period.
</ParamField>

<ParamField path="default_extensions" type="object">
  Default SSH certificate extensions to include if none are provided at signing time. Common extensions: `permit-pty`, `permit-port-forwarding`, `permit-agent-forwarding`.
</ParamField>

<ParamField path="allowed_extensions" type="string">
  Comma-separated list of extensions clients may request. Set to `*` to allow any. Empty means no overrides allowed.
</ParamField>

<ParamField path="algorithm_signer" type="string" default="default">
  Signing algorithm. Options: `ssh-rsa`, `rsa-sha2-256`, `rsa-sha2-512`, `default`.
</ParamField>

<ParamField path="not_before_duration" type="string" default="30s">
  Duration by which to backdate the certificate's valid-after field to account for clock skew.
</ParamField>

<ParamField path="allow_empty_principals" type="boolean" default="false">
  Allow issuing certificates with no principals (valid for any user). Not recommended.
</ParamField>

### Signing with specific principals

```bash
vault write ssh-client-signer/sign/my-role \
  public_key=@$HOME/.ssh/id_rsa.pub \
  valid_principals=ubuntu
```

### Signing a host key

To have Vault sign a host's public key (for `HostCertificate` in `sshd_config`):

```bash
# Create a host role
vault write ssh-host-signer/roles/hostrole \
  key_type=ca \
  ttl=87600h \
  allow_host_certificates=true \
  allowed_domains="example.com" \
  allow_subdomains=true

# Sign the host key
vault write ssh-host-signer/sign/hostrole \
  cert_type=host \
  public_key=@/etc/ssh/ssh_host_rsa_key.pub
```

## OTP mode

In OTP mode, Vault generates a random one-time password for the target host. A Vault SSH Helper daemon on the target host validates the OTP against Vault before allowing the SSH connection.

<Note>
  OTP mode requires installing `vault-ssh-helper` on every target host. CA mode is simpler to operate at scale.
</Note>

<Steps>
  <Step title="Enable the SSH engine">
    ```bash
    vault secrets enable -path=ssh ssh
    ```
  </Step>

  <Step title="Create an OTP role">
    ```bash
    vault write ssh/roles/otp-role \
      key_type=otp \
      default_user=ubuntu \
      cidr_list=10.0.0.0/8
    ```
  </Step>

  <Step title="Install vault-ssh-helper on target hosts">
    Download and configure `vault-ssh-helper` on each target host. Configure `/etc/vault-ssh-helper.d/config.hcl`:

    ```hcl
    vault_addr = "https://vault.example.com:8200"
    tls_skip_verify = false
    ssh_mount_point = "ssh"
    allowed_roles = "*"
    ```

    Update `/etc/pam.d/sshd` to use the helper for authentication.
  </Step>

  <Step title="Request an OTP and connect">
    ```bash
    # Request OTP
    vault write ssh/creds/otp-role ip=10.0.0.5
    # Response contains: key=<otp_value>

    # Connect using the OTP as the password
    ssh -o StrictHostKeyChecking=no ubuntu@10.0.0.5
    # Enter the OTP when prompted for a password
    ```
  </Step>
</Steps>

### OTP role parameters

<ParamField path="key_type" type="string" required>
  Must be `otp` for OTP mode.
</ParamField>

<ParamField path="default_user" type="string" required>
  The default username for which the OTP is generated.
</ParamField>

<ParamField path="cidr_list" type="string">
  Comma-separated list of CIDR blocks for which the role applies.
</ParamField>

<ParamField path="exclude_cidr_list" type="string">
  Comma-separated list of CIDR blocks excluded from this role.
</ParamField>

<ParamField path="port" type="number" default="22">
  SSH port. Informational only for OTP type.
</ParamField>
